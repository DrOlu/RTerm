# v1.9.1 Release Notes

## English

v1.9.1 adds **Advanced Automation** to RTerm: event-driven triggers, DAG/parallel playbooks, parameterized runbooks with secrets, idempotent desired-state config, approval-gated auto-remediation, and cross-host orchestration variables — all fully unit-tested (75 new tests, 909 total) and wired into the agent, both runtimes, and the settings store.

### What's new

**Event-driven triggers (`manage_trigger`).** Define triggers that fire a playbook (or propose a MOP change) when a condition matches:
- **Pattern** — a substring/regex seen in terminal output (e.g. `BGP.*DOWN` on a router group).
- **Threshold** — a monitor metric crossing a bound (e.g. `cpuUsagePercent > 90`).
- **Webhook** — fired manually (e.g. by CI) via the gateway.
Safety rails built in: per-trigger **cooldown** (default 300s) and a global **concurrency cap** so a noisy log or flapping metric can't spawn a remediation storm. Triggers persist to the automation store, are CRUD-managed via the `manage_trigger` agent tool, and every firing is recorded (reason + outcome) in an auditable fire log.

**DAG / parallel playbooks.** Steps can declare `dependsOn` (a dependency graph) instead of strictly linear order. The engine topologically sorts steps, runs independent steps in parallel (capped by the playbook's `maxParallelSteps`), and detects dependency **cycles** and **missing dependency ids** before touching any target. Fan-out/fan-in workflows (run step A on many, then join) now compose cleanly.

**Parameterized runbooks with secrets.** Playbooks declare `params[]` (name, default, `secret` flag). Values inject at run time (`run_playbook … paramValues={...}`) and substitute into step commands as `{{param.name}}`. Secret params are **masked** in run records/logs while still being used in commands.

**Idempotent desired-state config.** A step can declare `desiredState: { command, expect }` — if the check passes before running, the step is **skipped** (already in desired state). Re-running the playbook is a no-op, enabling converge-style automation instead of blind re-execution.

**Cross-host orchestration variables.** A step can `captureVar` (regex/substring) from its output into a run-scoped variable map, which later steps reference as `{{var}}` — e.g. capture an IP from step 1 and ping it in step 2.

**Approval-gated auto-remediation.** Threshold/pattern triggers with `actionType: propose-change` auto-propose a MOP change record when a fault is detected — the agent (or a human) approves, and the playbook executes with validation + automatic rollback, all recorded in the change ledger.

### Architecture

New modules (all pure + injectable + unit-tested):
- `automation/triggerEngine.ts` — trigger matching, cooldown/concurrency, fire log
- `automation/triggerRuntime.ts` — wires the engine to terminal data + monitor snapshots
- `automation/dagScheduler.ts` — topological ordering, cycle/missing-dep detection, parallel batches
- `automation/runbookEngine.ts` — param resolution, `{{var}}` substitution, secret masking, capture, desired-state
- `automation/orchestratedPlaybookRunner.ts` — the DAG/param/idempotent execution engine (linear playbooks keep the proven sequential runner; advanced playbooks route here automatically via `playbookNeedsOrchestration`)

Wiring: `manage_trigger` agent tool (schema + handler + description), `triggerEngine` on the tool context, runtime construction in both `startGyBackend` and `startElectronMain`, and `triggers[]` persistence in `AutomationManager`.
