# v1.9.2 Release Notes

## English

v1.9.2 brings **Advanced Automation to the standalone `rterm-backend` CLI** (published to npm) and adds a **NATS event mesh** so event-driven triggers fire **fleet-wide**, not just on the local host.

### `rterm-backend` now ships Advanced Automation

The standalone npm package (`npm i -g rterm-backend`, v1.9.2) includes the full Advanced Automation pillar from v1.9.1 — event-driven triggers, DAG/parallel playbooks, parameterized runbooks with secrets, idempotent desired-state config, and cross-host orchestration variables — verified to boot headless and serve its gateway.

### NATS event mesh (fleet-wide triggers)

New: point any RTerm / rterm-backend instance at a [NATS](https://nats.io) server and its terminal output + monitor snapshots publish onto shared subjects — so a pattern or threshold trigger fires **across the whole fleet** of connected instances, not just the local host.

- New `automation/natsEventBus.ts` — a thin, injectable transport adapter over `@nats-io/transport-node` (v3). Subjects (configurable prefix, default `rterm`): `rterm.term.data`, `rterm.monitor.snapshot`, `rterm.trigger.fire`. Payloads are JSON; subscriptions are live immediately (no reliance on iterator start), and `createTriggerRuntime` exposes a `busReady` promise so callers know when the mesh subscriptions are live.
- **Best-effort by design:** a NATS outage never blocks local automation — the bus is optional, and connection failures are logged and ignored.
- Configure in `settings.json`:

```jsonc
{ "nats": { "enabled": true, "url": "nats://localhost:4222", "prefix": "rterm" } }
```

Run a NATS server with the single-binary `nats-server` as the mesh hub.

### What it's for

- **Distributed remediation:** "watch `BGP.*DOWN` across *all* rterm-backend daemons and fire a central recovery playbook."
- **Fleet-wide SLO:** a `cpuUsagePercent > 90` threshold trigger that reacts to any host in the mesh.
- **Fan-out triggers:** one CI webhook fire drives playbooks on every connected instance.
