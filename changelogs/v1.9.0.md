# v1.9.0 Release Notes

## English

v1.9.0 is a feature release: **RTerm is now available as a standalone, installable CLI** — `rterm-backend` on npm — so you can run RTerm-as-a-service anywhere Node runs, with **no desktop app and no repo clone required**.

```bash
npm install -g rterm-backend
gybackend          # boots the full RTerm backend + WebSocket gateway
```

### What's new

- **`rterm-backend` npm package.** A single near-self-contained Node bundle that boots RTerm's entire backend — the AI agent, SSH/WinRM/Serial/local terminals, fleet orchestration, scheduled automation, and change management (MOP) — and serves it over the WebSocket JSON-RPC gateway (default `ws://0.0.0.0:17888`). Install globally or run via `npx rterm-backend`.
- **Standalone esbuild bundle.** New `apps/gybackend/build-standalone.mjs` produces `dist-standalone/gybackend.js`: ESM output (required — the backend's `betterSqlite3Runtime` uses `createRequire(import.meta.url)`, which is undefined under CJS), with `@gyshell/shared` and all pure-JS deps inlined. Only native/wasm packages stay external (`better-sqlite3`, `node-pty`, `serialport`, `ssh2`/`cpu-features`, tree-sitter wasm). A collision-free `globalThis` CJS-globals banner lets bundled CJS packages (ws, socks, ssh2 JS parts) call `require()` without redeclaring `createRequire`.
- **New npm scripts** in the gybackend workspace: `build:standalone` and `start:standalone`.

### How it works

The CLI prints its gateway endpoint on boot (`[gybackend] WebSocket RPC endpoint: ws://…`). Drive it exactly like the desktop app's gateway — same 72 RPC methods, same envelope (`{type:"gateway:response", id, ok, result|error}`), localhost skips token auth. Point `GYBACKEND_DATA_DIR` at the desktop app's settings directory to reuse your saved connections and model keys, or let it boot with a fresh data dir. See the [`rterm-gateway` skill](https://github.com/DrOlu/agent-skills/tree/main/skills/rterm-gateway) for the full RPC reference and client examples (Node CLI, Python, websocat).
