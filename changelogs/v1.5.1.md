# v1.5.1 Release Notes

## English

v1.5.1 is a focused drag-and-drop release for the file system panel. The headline addition is **native file drag-out** — you can now drag files from a Local terminal's file system panel straight onto the Finder/desktop or into any other app, mirroring the drag-in support GyShell already had. This release also fixes a regression introduced by the Electron 42 upgrade in v1.5.0: dragging files *from* the OS *into* the app silently stopped working because Electron 42 removed the legacy `File.path` property. Native drops are now routed through the supported `webUtils.getPathForFile` bridge again across the chat composer, terminals, and the file system panel. Finally, the `@` mention chips that were polished in v1.5.0 now truncate consistently in rendered messages too.

### Highlights You Will Notice

- **Drag files out to the OS (Finder / desktop / other apps)**
  - file rows in a **Local** terminal's file system panel are now draggable straight out of GyShell — drop them on the desktop, into Finder, or onto any app that accepts files, and they are copied out as real OS files
  - multi-select is supported: select several files and drag the group out in one gesture
  - the drag shows the file's real OS icon (with a neutral fallback) so it feels like dragging from Finder
  - this is symmetric with the existing drag-*in* support, so the file system panel now works in both directions
  - remote (SSH) entries are intentionally excluded from native drag-out — their bytes are not on this machine — while in-app drags (panel→panel, panel→terminal, panel→chat) keep working as before
- **Native file drops back into the app work again (Electron 42 fix)**
  - the v1.5.0 upgrade to Electron 42 removed the renderer's legacy `File.path`, which silently broke dragging files from the OS into GyShell — the dropped paths came back empty
  - drops are now resolved through Electron's supported `webUtils.getPathForFile` bridge, restored across the **chat composer**, **terminals**, and the **file system panel** (the panel's native-upload drop was the last one still on the old path and is now fixed too)
- **Consistent `@` mention chips in rendered messages**
  - long `@` mentions are now truncated to a readable length with a full-text tooltip not just in the composer but in **rendered chat messages and the queue** as well, so a long file/skill/terminal mention no longer stretches a bubble
  - the chip truncation logic is shared from a single helper and the badge styling was unified (letter-spacing/wrapping) so chips look the same everywhere

### Under the Hood

- **Unified native-path resolution**
  - a single `resolveNativeFilePath` helper (resolver-first, with a safe fallback to the legacy `path`) and a shared `getNativeFilePathResolver()` now back every native drop site, replacing three near-identical inline `window.gyshell.system.getPathForFile` probes
  - the file system panel dropped its bespoke `getNativeDropFilePaths` in favor of the shared `extractNativeDropFilePaths`, so all three drop targets behave identically
- **Native drag-out plumbing**
  - the preload exposes a new `system.startFileDrag(paths)` (the app→OS counterpart of `system.getPathForFile`), which sends an IPC the main process turns into `webContents.startDrag` with a resolved file icon and a neutral fallback so the drag never throws on macOS
- **Mention display extracted**
  - `truncateMentionDisplayText` / `MENTION_DISPLAY_CHAR_LIMIT` moved into a shared `mentionDisplay.ts` module reused by both the composer and the message renderer
- **Tests**
  - new extreme specs cover the native-path resolver (resolver precedence, throwing/empty fallback) and `getNativeFilePathResolver` (bridge present / missing), and the mention-parser spec now asserts the truncation + full-text tooltip on rendered badges

### What This Means in Practice

- The file system panel is now fully bidirectional: drag files in from your OS, and drag local files back out to the desktop or another app.
- Dragging files from Finder/Explorer into the chat, a terminal, or the file panel works again after the Electron 42 upgrade.
- Long `@` mentions stay tidy everywhere — in the input box and in the conversation history.

## 中文

v1.5.1 是一次聚焦于文件系统面板拖拽的更新。最大的新增是 **原生文件拖出**——你现在可以把文件从「本地（Local）」终端的文件系统面板直接拖到 Finder/桌面或任意其它应用里，与 GyShell 原有的拖入能力形成对称。本次更新还修复了 v1.5.0 升级 Electron 42 引入的一个回归：把文件*从*操作系统拖*入*应用会悄悄失效，因为 Electron 42 移除了旧的 `File.path` 属性。现在聊天输入框、终端和文件系统面板的原生拖入都重新走 Electron 支持的 `webUtils.getPathForFile` 桥接。最后，v1.5.0 打磨过的 `@` 提及标签现在在已渲染的消息里也会一致地截断。

### 你会直接感知到的升级

- **把文件拖出到操作系统（Finder / 桌面 / 其它应用）**
  - 「本地」终端文件系统面板里的文件行现在可以直接拖出 GyShell——拖到桌面、拖进 Finder、或拖到任意接受文件的应用上，文件会作为真实的 OS 文件被复制出去
  - 支持多选：选中多个文件后可以一次性把整组拖出
  - 拖拽时会显示文件真实的系统图标（并带中性回退图标），手感与从 Finder 里拖拽一致
  - 这与原有的拖*入*能力对称，因此文件系统面板现在双向可用
  - 远程（SSH）条目被有意排除在原生拖出之外——它们的字节不在本机——而应用内拖拽（面板→面板、面板→终端、面板→聊天）保持原样可用
- **从操作系统拖文件回应用重新可用（Electron 42 修复）**
  - v1.5.0 升级到 Electron 42 移除了渲染层旧的 `File.path`，这悄悄破坏了把文件从操作系统拖入 GyShell 的能力——拖入得到的路径变成了空
  - 现在拖入通过 Electron 支持的 `webUtils.getPathForFile` 桥接来解析，已在 **聊天输入框**、**终端** 和 **文件系统面板** 三处恢复（面板的原生上传拖放是最后一处仍走旧路径的地方，现在也一并修复）
- **已渲染消息中的 `@` 提及标签更一致**
  - 较长的 `@` 提及现在不仅在输入框、也在 **已渲染的聊天消息与队列** 里被截断到易读长度并带完整文本 tooltip，因此长文件/技能/终端提及不再撑开气泡
  - 标签截断逻辑由单一辅助函数共享，徽标样式（字间距/换行）也做了统一，让标签在各处外观一致

### 底层改动

- **统一的原生路径解析**
  - 单一的 `resolveNativeFilePath` 辅助函数（resolver 优先，并安全回退到旧的 `path`）与共享的 `getNativeFilePathResolver()` 现在支撑所有原生拖放点，替换掉三处几乎相同的内联 `window.gyshell.system.getPathForFile` 探测
  - 文件系统面板移除了自有的 `getNativeDropFilePaths`，改用共享的 `extractNativeDropFilePaths`，让三个拖放目标行为完全一致
- **原生拖出管线**
  - preload 新增 `system.startFileDrag(paths)`（`system.getPathForFile` 的应用→OS 对应物），它发送一条 IPC，由主进程转为 `webContents.startDrag`，并附带解析出的文件图标与中性回退图标，使拖拽在 macOS 上永不抛错
- **提及展示抽取**
  - `truncateMentionDisplayText` / `MENTION_DISPLAY_CHAR_LIMIT` 被移入共享的 `mentionDisplay.ts` 模块，供输入框与消息渲染器共用
- **测试**
  - 新增的 extreme 规格覆盖了原生路径解析器（resolver 优先、抛错/空值回退）与 `getNativeFilePathResolver`（桥接存在/缺失）；提及解析器的规格现在也会断言已渲染徽标上的截断与完整文本 tooltip

### 这次升级带来的实际体验

- 文件系统面板现在完全双向：从操作系统把文件拖进来，也能把本地文件拖回桌面或另一个应用。
- 在 Electron 42 升级之后，从 Finder/资源管理器把文件拖进聊天、终端或文件面板重新可用。
- 长 `@` 提及在各处都保持整洁——无论是输入框还是对话历史里。
