# v1.8.3 Release Notes

## English

v1.8.3 is a bug-fix release. It resolves a real connectivity bug where **SSH connections created without an explicit `authMethod` field failed to authenticate** — most notably terminals opened programmatically through the **RTerm WebSocket gateway** (`terminal:createTab`), and any saved/forwarded config where `authMethod` was absent. Connections to network gear (e.g. Cisco IOS-XE) and password/key hosts now authenticate correctly in these paths.

### The bug

`SSHConnectionConfig.authMethod` is typed as **required** (`'password' | 'privateKey'`), and the SSH backend's `spawn()` (and jump-host path) branched on it like so:

```ts
if (sshConfig.authMethod === "password") { /* set password */ }
else if (sshConfig.authMethod === "privateKey") { /* set key */ }
```

When a caller **omitted** `authMethod` (the gateway `terminal:createTab` path, and any config built without going through the Connections panel), **neither branch ran and no credential was ever set**, so the connection failed with:

```
✘ SSH Error: All configured authentication methods failed
```

The Connections-panel / `open_terminal_tab` path always populated `authMethod`, which is why interactive connections worked while gateway-driven ones did not — this looked, at first glance, like an ssh-rsa signature or keyboard-interactive negotiation problem. It was not: the SSH handshake (host key `rsa-sha2-512`) completed and the signature verified; the failure was purely that **no auth credential reached the ssh2 client**.

### The fix

`SSHBackend.spawn()` (and the jump-host connect path) now **infer the effective auth method when `authMethod` is missing**:

```ts
const effectiveAuthMethod =
  sshConfig.authMethod === "password" || sshConfig.authMethod === "privateKey"
    ? sshConfig.authMethod
    : sshConfig.privateKey || sshConfig.privateKeyPath
      ? "privateKey"
      : "password";
```

So a password or key supplied without an explicit `authMethod` is honoured instead of silently dropped.

### Verification

- **Reproduced first:** gateway `terminal:createTab` to a Cisco IOS-XE host failed with *"All configured authentication methods failed"*; direct `ssh2` with the correct password succeeded — isolating the bug to the missing-`authMethod` branch (not ssh-rsa/keyboard-interactive).
- **After the fix:** the exact gateway-style config (password present, `authMethod` absent) now resolves to `password` and connects — `show version` returned `Cisco IOS XE Software, Version 17.18.03a`.
- `npm run typecheck:node` — clean
- `npm run lint` — 0 errors
- `npm test` — 834 tests, 0 failures

### Note

This was **not** a credential or connectivity problem, and **not** a legacy ssh-rsa signature issue — the host key negotiated `rsa-sha2-512` and verified fine. It was a missing-credential branch in the SSH backend for configs without an explicit `authMethod`. A separate, unrelated pre-existing issue remains in the standalone `gybackend` bundle (a stale `packages/shared` module reference); it does not affect the desktop app's gateway or this fix.
