# v2.0.0 Release Notes

## English

v2.0.0 is a major feature release: **the SRE pillar**. RTerm gains a complete site-reliability-engineering foundation — metrics time-series, uptime watchdogs, SLO/SLI with error budgets and burn-rate, alertmanager-style alert routing, incident management with AI root-cause analysis and postmortems, golden-signal dashboards with capacity forecasting, synthetic blackbox checks, and config-drift detection — all fully unit-tested (33 new tests, 975 total) and built as pure, injectable modules composable with RTerm's existing monitor, triggers, NATS mesh, and MOP.

### Tier 1 — SRE core

- **MetricsLedger** — a time-series store for resource-monitor snapshots. Every snapshot flattens to a compact metric point (cpu/mem/disk-max/load/net/gpu/processes) per host, with ring-buffer retention, single-metric series, linear-trend slope (units/day), and **days-to-threshold forecasting** ("disk full in N days").
- **UptimeWatchdog** — liveness/heartbeat probes (tcp/ssh/http/command) per host with up/degraded/down state, consecutive-failure thresholds, per-target intervals, and a state-transition callback to raise alerts/incidents.
- **SloService** — SLO/SLI definitions with rolling windows, computed SLI, **error-budget remaining**, **burn rate** (actual vs. allowed error), and fast-burn alerting. Injected SLI source (good/total events) — feeds from synthetic checks or health probes.
- **AlertService** — alertmanager-style alert routing: fingerprint-based grouping, dedupe windows, maintenance **silences**, severity-threshold channels (telegram/webhook/email/slack via injected senders), and send history.

### Tier 2 — reliability operations

- **IncidentLedger** — auto-create incidents from signals, track open → mitigated → resolved with a full timeline, link evidence and the runbook playbook for that failure class, capture AI **root-cause analysis**, and auto-generate **postmortem** documents from timeline + RCA.

### Tier 3 — capacity & golden signals

- **GoldenSignals** — per-host saturation (cpu/mem/disk), traffic (net B/s), latency percentiles (p50/p95/p99 from synthetic series), and error rate, plus a **capacity forecast** table (current disk% + days-to-full per host).

### Tier 4 — advanced / fleet-scale

- **SyntheticChecks** — scripted blackbox probes (http/tcp/ssh/command) from any zone outpost on a schedule, recording latency + ok per check and feeding the SLO SLI source (good/total) and the golden-signal latency/error series.
- **DriftDetector** — periodically re-render a config template and diff it against the device's live config, with ignore-patterns, line-level added/removed diffs, drift callbacks, and optional **MOP auto-remediation**.

### Architecture

All eight modules are **pure + injectable** (no hard dependencies on TerminalService/MonitorService) in `packages/backend/src/services/sre/`, following RTerm's established pattern: storage injected (SQLite-wrap later), probe/sender functions injected, deterministic `now` for tests. They compose with the existing `ResourceMonitorService` (snapshots → MetricsLedger), `TriggerService` (fires → AlertService), NATS mesh (fleet-wide), and MOP (drift remediation).
